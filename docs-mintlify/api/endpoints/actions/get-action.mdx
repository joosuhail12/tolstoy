## Overview

Retrieves complete details for a specific action, including its configuration, input schema, execution statistics, and version history. This endpoint is essential for:

- Inspecting action configurations before execution
- Understanding input requirements and validation rules
- Reviewing execution statistics and performance metrics
- Debugging action setup and parameters
- Building dynamic UI for action selection and execution

## Authentication

<ParamField header="x-org-id" type="string" required>
  Your organization ID - determines which organization context to search in
</ParamField>

<ParamField header="x-user-id" type="string" required>
  Your user ID - used for permission checks and audit logging
</ParamField>

## Path Parameters

<ParamField path="id" type="string" required>
  The unique identifier of the action to retrieve. Can be either the action ID or the action key.
</ParamField>

## Examples

<CodeGroup>
```bash cURL
curl -X GET https://tolstoy.getpullse.com/actions/act_def456ghi789 \
  -H "x-org-id: org_abc123def456" \
  -H "x-user-id: user_xyz789abc"
```

```javascript JavaScript SDK
import { TolstoyClient } from '@tolstoy/sdk';

const tolstoy = new TolstoyClient({
  orgId: 'org_abc123def456',
  userId: 'user_xyz789abc'
});

const action = await tolstoy.actions.get('act_def456ghi789');

console.log(`Action: ${action.name}`);
console.log(`Method: ${action.method} ${action.endpoint}`);
console.log(`Executions: ${action.stats.totalExecutions}`);
console.log(`Success Rate: ${
  (action.stats.successfulExecutions / action.stats.totalExecutions * 100).toFixed(1)
}%`);
```

```python Python SDK
from tolstoy import TolstoyClient

client = TolstoyClient(
    org_id='org_abc123def456',
    user_id='user_xyz789abc'
)

action = client.actions.get('act_def456ghi789')

print(f"Action: {action.name}")
print(f"Method: {action.method} {action.endpoint}")
print(f"Executions: {action.stats.total_executions}")
success_rate = (action.stats.successful_executions / action.stats.total_executions * 100)
print(f"Success Rate: {success_rate:.1f}%")
```
</CodeGroup>

## Alternative Access Methods

### Get by Action Key

You can also retrieve an action using its unique key instead of ID:

```bash
GET /actions/stripe-create-customer
```

This is useful when you know the action key but not the ID.

### Response Format

Both ID and key lookups return identical response formats.

## Understanding Input Schema

The input schema defines what parameters the action accepts:

<Tabs>
  <Tab title="String Parameters">
    ```json
    {
      "name": "email",
      "type": "string",
      "required": true,
      "description": "Customer email address",
      "validation": {
        "format": "email",
        "maxLength": 255
      }
    }
    ```
    
    String parameters can include format validation (email, url, uuid) and length constraints.
  </Tab>
  
  <Tab title="Number Parameters">
    ```json
    {
      "name": "amount",
      "type": "number",
      "required": true,
      "description": "Transaction amount in cents",
      "validation": {
        "minimum": 50,
        "maximum": 1000000,
        "multipleOf": 1
      }
    }
    ```
    
    Number parameters support minimum/maximum values and multiple constraints.
  </Tab>
  
  <Tab title="Object Parameters">
    ```json
    {
      "name": "address",
      "type": "object",
      "required": false,
      "description": "Customer billing address",
      "validation": {
        "properties": {
          "street": {"type": "string", "required": true},
          "city": {"type": "string", "required": true},
          "zipCode": {"type": "string", "pattern": "^\\d{5}$"}
        }
      }
    }
    ```
    
    Object parameters can define nested schemas for complex data structures.
  </Tab>
  
  <Tab title="Array Parameters">
    ```json
    {
      "name": "tags",
      "type": "array",
      "required": false,
      "description": "Customer tags",
      "validation": {
        "minItems": 1,
        "maxItems": 10,
        "items": {
          "type": "string",
          "maxLength": 50
        }
      }
    }
    ```
    
    Array parameters can specify item types and quantity constraints.
  </Tab>
</Tabs>

## Execution Statistics

The `stats` object provides insights into action performance:

<AccordionGroup>
  <Accordion title="Success Rate">
    Calculate the success rate:
    ```javascript
    const successRate = (stats.successfulExecutions / stats.totalExecutions) * 100;
    console.log(`Success rate: ${successRate.toFixed(1)}%`);
    ```
  </Accordion>
  
  <Accordion title="Performance Metrics">
    Monitor execution performance:
    ```javascript
    if (stats.avgDurationMs > 5000) {
      console.warn('Action is running slower than expected');
    }
    ```
  </Accordion>
  
  <Accordion title="Usage Tracking">
    Track action popularity:
    ```javascript
    const executionsToday = await getExecutionsSince(new Date().setHours(0,0,0,0));
    console.log(`Executed ${executionsToday} times today`);
    ```
  </Accordion>
</AccordionGroup>

## Version Information

Actions support versioning for configuration changes:

- **Version Number**: Increments with each configuration change
- **Backward Compatibility**: Existing workflows continue using the version they were created with
- **Schema Evolution**: Input schemas can be extended without breaking existing workflows

## Best Practices

<AccordionGroup>
  <Accordion title="Schema Validation">
    - Always validate the input schema before building execution requests
    - Check `required` fields to ensure all necessary parameters are provided
    - Use `validation` rules to implement client-side validation
    - Handle `default` values appropriately in your UI
  </Accordion>
  
  <Accordion title="Performance Monitoring">
    - Monitor `avgDurationMs` for performance degradation
    - Track success rates to identify problematic actions
    - Use `lastExecutedAt` to identify unused actions
    - Set up alerts for actions with high failure rates
  </Accordion>
  
  <Accordion title="Access Control">
    - Implement role-based access to sensitive actions
    - Log access to actions with sensitive data
    - Use the `createdBy` field for audit trails
    - Consider action visibility based on user permissions
  </Accordion>
</AccordionGroup>
