## Overview

The Retry Execution endpoint allows you to restart a failed workflow execution, optionally with modified parameters or starting from a specific step. This is useful for handling transient failures or continuing from where an execution stopped.

## Authentication

<ParamField header="x-org-id" type="string" required>
  Organization ID for multi-tenant access control
</ParamField>

<ParamField header="x-user-id" type="string" required>

</ParamField>

<ParamField header="authorization" type="string" required>
  Bearer token for API authentication
</ParamField>

## Path Parameters

<ParamField path="id" type="string" required>
  The unique identifier of the execution to retry
</ParamField>

## Request Body

<ParamField body="fromStep" type="string">
  Step ID to resume execution from (defaults to failed step)
</ParamField>

<ParamField body="parameters" type="object">
  Override execution parameters for the retry
</ParamField>

<ParamField body="retryMode" type="string">
  Retry mode: "from_failure", "from_beginning", or "from_step" (defaults to "from_failure")
</ParamField>

<ParamField body="maxRetries" type="number">
  Maximum number of retry attempts for this execution (defaults to 3)
</ParamField>

<ParamField body="retryDelay" type="number">
  Delay in seconds before starting the retry (defaults to 0)
</ParamField>

<ParamField body="reason" type="string">
  Optional reason for retrying the execution
</ParamField>

## Retry Modes

### from_failure (Default)
- Resumes from the step that failed
- Maintains previous step results
- Most efficient for transient failures

### from_beginning  
- Restarts the entire workflow
- Clears all previous step results
- Use when early steps need correction

### from_step
- Resumes from a specific step ID
- Requires `fromStep` parameter
- Useful for debugging or partial re-execution

## Retry Limits

Each execution has retry limits to prevent infinite loops:
- **Default limit**: 3 retries per execution
- **Maximum limit**: 10 retries (for enterprise plans)
- **Retry counter**: Tracks attempts across all retry modes

## Parameter Overrides

When retrying, you can override original execution parameters:
- **Timeout values**: Adjust step timeouts
- **Retry settings**: Modify step-level retry behavior  
- **Input data**: Update workflow inputs
- **Environment**: Change execution environment

## State Preservation

During retries:
- **Success states**: Completed steps remain completed
- **Failed states**: Failed steps are reset to pending
- **Data**: Previous step outputs are preserved unless overridden
- **Context**: Execution context and variables are maintained

## Retryable States

Executions can only be retried when in these states:
- `failed` - Execution failed and can be retried
- `canceled` - Manually canceled execution can be retried

Cannot retry executions in these states:
- `completed` - Successfully finished executions
- `running` - Currently executing
- `pending` - Not yet started

## Automatic vs Manual Retries

### Automatic Retries
- Configured per step in workflow definition
- Trigger immediately on step failure
- Use exponential backoff delays
- Limited to step-level retry counts

### Manual Retries
- Triggered via this API endpoint
- Allow human intervention and analysis
- Support custom parameters and modes
- Subject to execution-level retry limits

## Webhook Events

Retry operations trigger webhook events:
- `execution.retry_started` - When retry execution begins
- `execution.retry_completed` - When retry succeeds
- `execution.retry_failed` - When retry also fails
